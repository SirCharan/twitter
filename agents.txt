## Databases

Two separate SQLite databases. Same schema pattern, different paths.

| DB | Used by | Path env var |
|----|---------|-------------|
| `stocky.db` | TG bot | `DB_PATH` (default: stocky.db) |
| `stocky_web.db` | Web backend | `DB_PATH` (default: stocky_web.db) |

### Tables (both DBs have similar structure)

| Table | Key columns | Purpose |
|-------|------------|---------|
| `kite_session` | access_token, login_time | Cache Kite auth token (single row, id=1) |
| `web_users` | username, password_hash | Web app login (web DB only) |
| `conversations` | session_id, role, content, message_type, structured_data | Chat history |
| `pending_actions` | action_id, action_type, action_data, status | Trade confirmation queue |
| `alerts` | symbol, target_price, direction, triggered | Price alerts |
| `trade_history` | order_id, symbol, txn_type, qty, price, status | Order log |
| `command_log` | command, args, source, ts | Usage tracking |
| `api_call_log` | service, endpoint, tokens, ts | Groq token/cost tracking |

## AI Model Routing

### Telegram Bot (stocky-ai/bot/handlers/nlp.py)

```
User message
    │
    ▼
_parse_natural()          ← regex NLP, zero AI cost
    │ match → dispatch to handler
    │ no match ↓
    ▼
_CHAT_RE match            ← greetings, "what is X", "explain Y"
AND no _TRADING_TERMS     ← NOT buy/sell/nifty/price/portfolio etc.
    │ yes → chat_basic()  ← llama-3.1-8b-instant (~0.5-1s)
    │ no  ↓
    ▼
_ai_fallback()
    → interpret_intent()  ← llama-3.3-70b-versatile (~1-3s)
    → dispatch to handler
```

### Web Backend (stocky-web/backend/app/handlers/chat.py)

```
POST /api/chat
    │
    ▼
_parse_natural()          ← same regex NLP
    │ match → dispatch
    │ no match ↓
    ▼
interpret_intent(text, history=last_10_msgs)
    → llama-3.3-70b-versatile
    → {intent, args, reply}
    → dispatch to handler
```

## Broker: Zerodha Kite Connect v5

- Auto-login runs at startup via TOTP (`pyotp`)
- Scheduler re-runs auto-login at 7:40 AM IST (before market open)
- **One active session per API key** — whichever app called auto_login last holds the token
- Token cached in `kite_session` table (row id=1, overwritten each login)
- Both TG bot and web backend share the same Kite credentials

## Critical Rules for AI Agents

1. **Landing page changes**: ALWAYS push to `SirCharan/stocky-landing` separately. See `CLAUDE.md`.
2. **Kite session**: Don't add a second login call — it will break the other app's session.
3. **Two databases**: TG bot and web backend have separate DBs. Don't cross-reference them.
4. **Token display multiplier**: `/usage` shows actual tokens × 101. This is intentional branding.
5. **Cost formula**: `(tokens × 0.4 × $15 + tokens × 0.6 × $75) / 1M` — Claude Opus 4.6 pricing for display purposes, even though underlying model is Groq Llama.
6. **`profit_margin`** in `_get_fundamental_data` is already stored as a percentage (multiplied by 100) in the result dict — don't multiply again in frontend.

## Common Patterns

- TG bot handlers are decorated with `@authorized` (from `bot/auth_check.py`)
- Web handlers use `get_current_user()` FastAPI dependency (JWT)
- Both use identical Kite auth logic — compare `kite_auth.py` in both dirs if you see auth issues
- `context.args` in TG bot holds command arguments as a list of strings
- Web backend `interpret_intent()` passes last 4 messages as context (not full 10 — keeps intent parsing fast)
- `analyse_verdict()` uses max_tokens=128 intentionally — verdict must be short
